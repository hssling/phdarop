# Architecture

PHDAROP is a Next.js (App Router) application whose business logic is a set of **deterministic
analytical engines**. The web layer is thin; the value is in `lib/phda/`.

## Request lifecycle

1. **UI** (`app/page.tsx`) collects a question and `POST`s it to `/api/analyze`.
2. **Route** (`app/api/analyze/route.ts`, Node runtime):
   - `lib/ai/extract.ts` → `ExtractedRequest` (analysis type, domain, budget, workforce,
     equity emphasis). Uses the AI Gateway if configured; otherwise a deterministic
     keyword classifier (`heuristicExtract`).
   - `lib/phda/pipeline.ts` → runs the relevant engine(s) and assembles a `Report`.
   - `lib/ai/narrative.ts` → optionally rewrites the executive summary, **grounded only on
     computed facts**. No-op without AI.
   - `lib/phda/report.ts` → serializes a downloadable Markdown rendering.
3. **UI** renders the `Report` (`components/ReportView.tsx`) with KPIs, tables, recommendations.

## Engines (`lib/phda/`)

| File | Responsibility | Mirrors research engine |
|---|---|---|
| `mcda.ts` | AHP/BWM/TOPSIS/PROMETHEE + consensus + Spearman | E01, E08 |
| `optimization.ts` | LP allocation, status-quo comparison, frontier, stability | E09 |
| `attribution.ts` | PAF, attributable deaths/DALYs/cost | E06 / Part 4 |
| `indices.ts` | PPGI, IFI, AII, RMI, PBI, HEDI | Part 3 |
| `datasets.ts` | Calibrated demo scenarios (→ MADB in production) | E04 |
| `pipeline.ts` | Orchestration + report assembly | Part 7 |
| `report.ts` | Markdown serialization + standard limitations | E13 |
| `types.ts` | Shared types (mirror the notation register) | formulas/notation.md |

## Determinism boundary

```
   AI (optional, never produces numbers)        Deterministic (always runs)
   ───────────────────────────────────────      ──────────────────────────────────
   extract.ts   : question → parameters    →     pipeline + engines → numbers
   narrative.ts : numbers → prose          ←     (LP, MCDA, PAF, indices)
```

If AI is disabled or fails, both arrows fall back to deterministic implementations. The
numbers a minister sees are **never** generated by a language model.

## Runtime & scaling

- **Node.js runtime** (the LP solver is a Node package). `maxDuration` 120s.
- Stateless — horizontally scalable on Vercel Fluid Compute; no DB needed for the demo core.
- In production, `datasets.ts` is replaced by connectors to the Master Analytical Database
  (lakehouse + PostGIS), per [docs/engines/E12-software-factory.md](engines/E12-software-factory.md).

## Production hardening checklist

- Swap calibrated scenarios for governed data sources (HMIS/NFHS/GBD/programme).
- Move point-estimate optimization to stochastic/robust (propagate the Bayesian posterior).
- Add auth (Clerk/Auth0), rate limiting (BotID/firewall), and run provenance (MLflow).
- Persist reports (Vercel Blob / Postgres) and add audit logging for policy traceability.
